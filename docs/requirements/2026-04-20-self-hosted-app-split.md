# 自建 App 消息展示与前后端拆分需求

## 背景

当前仓库只有一个用于接收企业微信推送的 Flask 服务。由于企业微信相关接入路径受备案等现实约束，目标调整为自建 App：保留后端消息接收能力，同时新增一个使用 uni-app x 编写的移动端应用，用于在 Android 与 iOS 上展示已经推送的消息。

## 目标

1. 将现有后端代码整体收敛到 `backend/` 目录。
2. 在 `backend/` 中保留现有发送能力，并新增消息持久化能力。
3. 在 `backend/` 中新增供 App 读取消息的 HTTP API。
4. 新建 `frontend/` 目录，使用 uni-app x 项目骨架实现消息列表与详情展示。
5. 根目录保留统一入口文档与 Docker Compose，能够继续启动后端。

## 交付物

- `backend/`：Python Flask 服务、Dockerfile、requirements、测试。
- `frontend/`：uni-app x 项目骨架、消息列表页、消息详情页、请求封装与配置文件。
- 根目录文档与 compose 更新。
- vibe 运行产物与验证记录。

## 约束

- 默认语言使用简体中文。
- 沿用当前 Python/Flask 技术栈，不引入重型后端框架。
- 前端使用 uni-app x，不切换到其他跨端方案。
- 不在本轮实现原生系统级推送通道接入，例如 APNs、FCM、厂商推送 SDK。
- 不要求本轮完成 App 上架、签名、证书与发布流程。
- 不引入数据库，后端先使用本地 JSON 文件完成消息持久化。

## 验收标准

1. 推送接口仍可通过 `/send/{key}` 接收文本、Markdown、图片请求。
2. 每次成功推送后，后端会落盘保存一条可供 App 展示的消息记录。
3. App 可调用后端接口查看消息列表和单条详情。
4. 仓库结构拆分为 `backend/` 与 `frontend/` 后，后端测试仍通过。
5. 根目录部署说明能指导服务端启动后端，并说明前端如何用 HBuilderX/uni-app x 运行。

## 产品验收标准

- 打开 App 后可以看到消息列表。
- 点击列表项可以看到消息详情。
- 支持手动下拉刷新和基础轮询刷新。
- 消息至少展示标题、类型、接收人、创建时间、正文。

## 人工抽检

1. 启动后端后调用 `/send/{key}` 发送一条文本消息。
2. 调用 `/api/messages` 能看到新增记录。
3. 用 uni-app x 打开前端项目，配置后端基地址后进入首页。
4. 首页能拉取消息列表，详情页能展示正文。

## 完成语言策略

- 只有在测试或显式验证通过的范围内，才使用“已验证”表述。
- 未执行的移动端真机构建、Docker 实构建、APNs 相关流程必须标记为未验证。

## 交付真实性约定

- 不伪造真机运行结果。
- 不伪造 uni-app x 编译结果。
- 不伪造 Docker 镜像构建结果。

## 非目标

- 系统推送通知、离线推送、角标同步。
- 用户登录体系、多用户隔离、消息删除权限。
- 数据库存储与后台管理界面。

## 自主执行模式

- 采用受控但尽量自主的实现方式。
- 在不阻断进度的前提下，使用高置信假设推进 MVP。

## 推断假设

1. 当前阶段的“通过 App 展示推送消息”指的是 App 拉取并展示服务端已保存的消息，而不是系统级通知通道。
2. 移动端暂不做登录，默认可直接读取消息列表。
3. uni-app x 项目以 HBuilderX 可打开的最小骨架为目标，不在当前环境中强求完整构建验证。
